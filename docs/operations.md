# Trace View Core Operations and Governance

## 1. 문서 목적

이 문서는 Trace View Core의 운영 정책과 거버넌스를 정의한다.
목표는 문서 생성, 검증, 승인, 민감정보 처리, 변경 관리의 책임을 명확히 하는 것이다.

## 2. 역할과 책임

### 2.1 Main Agent

- 전체 문서 구조와 우선순위 조율
- 단계 상태 확정
- 최종 통합 문서 반영

### 2.2 생성 에이전트

- PRD, 설계서, 화면 문서 등 초안 생성
- Open Questions와 Decision Log 기록

### 2.3 Validation Agent

- 논리적 모순 확인
- 누락 요구사항 점검
- 애매한 표현과 과도한 가정 식별
- Must Fix와 Recommended Improvements 분리

### 2.4 Status Agent

- 진행 현황 집계
- 단계별 상태와 리스크 보고
- 수시 상태 보고 문서 갱신

### 2.5 메타데이터 승인자

- 자동 생성 설명 승인 또는 반려
- 수동 주석 승인
- 운영 메모 최종 확정

## 3. 문서 상태 정의

아래 상태는 `문서 승인 상태`를 의미한다.
트랙 또는 작업 진행 상태는 `planning-design-track-board.md`, `status-report.md`에서 별도로 관리한다.

- Draft: 초안 작성 상태
- In Progress: 작성 또는 수정 중
- Review Pending: 검증 대기 상태
- Needs Revision: 검증 결과 수정 필요
- Validated: 검증 완료
- Approved: 최종 승인 완료
- On Hold: 선행 이슈 대기
- Archived: 폐기 또는 이력 보관

## 4. 문서 생성 및 검증 흐름

1. 생성 에이전트가 초안을 작성한다.
2. 메인 에이전트가 구조와 범위를 1차 점검한다.
3. Validation Agent가 별도 리뷰를 수행한다.
4. Must Fix는 반영 후 재검증한다.
5. 통과 시 문서 상태를 Validated로 변경한다.
6. 최종 승인 필요 문서는 Approved로 승격한다.

## 5. 검증 기준

- 상위 문서와 모순이 없는가
- MVP 범위를 넘어서는 과도한 설계가 없는가
- 측정 기준과 운영 책임이 명시되었는가
- 자동 추론 결과의 근거와 한계가 표현되었는가
- 보안 및 민감정보 정책이 반영되었는가

## 6. 승인 기준

- Must Fix가 모두 처리되었는가
- Open Questions가 명확히 분리되었는가
- 산출물이 다음 단계 입력으로 사용 가능한 수준인가
- 관련 리뷰 문서가 연결되어 있는가

## 7. 민감정보 처리 정책

- Authorization 헤더, 세션 토큰, 개인정보는 저장 전 제거 또는 마스킹
- 운영 환경 데이터는 최소 수집 원칙 적용
- 메모와 주석 접근은 권한 기반으로 제한
- 스냅샷에는 민감 원본 데이터를 저장하지 않음

## 8. 접근 권한 정책

- 조회 권한: 기본 팀 구성원
- 주석 수정 권한: 담당 도메인 개발자 이상
- 승인 권한: 도메인별 지정 리드
- 운영 설정 변경 권한: 운영 관리자

## 9. 자동 요약 운영 정책

- 모든 자동 생성 요약은 Draft로 시작
- 승인 전에는 참고 정보로만 노출
- Rejected 사유는 기록
- 승인된 설명만 기본 요약으로 표시

## 10. 분석 재실행 정책

- 대표 릴리즈 전 1회 이상 재분석
- 구조 변경이 큰 경우 수동 재분석
- 스냅샷은 코드 버전과 함께 저장
- MVP의 런타임 추적 범위는 개발 및 스테이징 환경으로 제한

## 11. 감사 이력 정책

- 검증 결과 문서 보관
- 승인 및 반려 이력 보관
- 문서 변경 이력 유지
- 의사결정 로그 누적 관리

## 12. 리스크 관리

- 검증 누락 시 문서를 Approved로 올리지 않음
- 민감정보 처리 정책 미적용 시 런타임 수집 비활성화
- 승인자 부재 시 Approved 승격 금지

## 13. Open Questions

- 운영 환경 추적을 향후 어떤 조건에서 허용할지
