# Trace View Core PoC Plan

## 1. 문서 목적

이 문서는 Trace View Core의 PoC 범위, 검증 방법, 단계별 실행 계획을 정의한다.
목표는 MVP 이전에 핵심 가설을 낮은 비용으로 검증하는 것이다.

## 2. PoC 목표

- 대표 화면 기준으로 화면-API 매핑이 실무에서 유용한지 검증
- API 기준으로 서버 진입점과 서비스 체인 추적이 가능한지 검증
- 신규 인력이 기존 방식보다 빠르게 맥락을 파악하는지 검증
- 문서 생성과 검증 운영 비용이 과도하지 않은지 검증

## 3. PoC 범위

포함:

- 대표 화면 20~30개
- 단일 프론트 애플리케이션
- 단일 백엔드 서비스 또는 BFF
- 검색, 상세, 그래프의 기본 흐름
- 수동 주석과 검증 리뷰

제외:

- 멀티 리포지토리 통합
- 배치 및 이벤트 흐름 완전 추적
- 운영 환경 전체 커버리지
- 변경 위험도 점수 자동 계산

## 4. 검증 가설

### 가설 1

신규 인력은 화면명만으로 관련 API를 기존 방식보다 빠르게 찾을 수 있다.

### 가설 2

개발자는 API 기준으로 핵심 서비스 로직 위치를 일관되게 찾을 수 있다.

### 가설 3

정적 분석 결과에 수동 보강을 결합하면 온보딩 문서보다 최신성이 높다.

### 가설 4

검증 에이전트 기반 문서 리뷰는 실무 문서 품질을 안정화한다.

## 5. 성공 기준

- 화면 기준 API 탐색 시간이 기준선 대비 감소
- API 기준 서비스 로직 탐색 시간이 기준선 대비 감소
- 대표 화면군에서 실무 사용 가능한 커버리지 확보
- 문서 검증 후 Must Fix 누락 없이 상태 관리 가능

## 6. 측정 설계

대상:

- 신규 인력 3명 이상 또는 준하는 실험 대상
- 기능 담당 개발자 2명 이상

측정 과제:

- 특정 화면의 관련 API 찾기
- 특정 API의 핵심 서비스 메서드 찾기
- 변경 대상 서비스의 연결 화면 찾기

측정 방식:

- 동일 과제를 도구 도입 전과 후에 수행
- 시작 시각, 종료 시각, 실패 여부 기록
- 보조 메모로 혼란 지점 기록

## 7. 단계별 실행 계획

### Step 1. 대상 선정

- 대표 화면군 선정
- 대표 API군 선정
- 연동할 백엔드 서비스 선정

### Step 2. 문서 및 설계 확정

- PRD Validated
- Architecture Validated
- 데이터 모델, 파이프라인, 운영 정책 정리

### Step 3. PoC 설계 검증

- Validation Agent 리뷰
- `open-questions.md` 기준 Open Questions 정리
- 측정 시나리오 확정

### Step 4. 구현 착수 준비

- 구현 범위 고정
- 기술 스택 선택
- 일정 추정

## 8. 리스크

- 대표 화면 선정이 편향될 수 있음
- 프레임워크 구조에 따라 정적 분석 난이도 차이가 큼
- 측정 대상 수가 적으면 결과 해석이 불안정할 수 있음

## 9. 대응 전략

- 화면군을 난이도별로 나눠 선정
- 정적 분석과 수동 보강을 병행
- 측정 시나리오를 표준화

## 10. 인수 조건

PoC 준비 단계 완료 조건은 아래와 같다.

- 핵심 기획 문서가 Validated 상태
- 설계 문서가 Validated 상태
- 측정 시나리오와 성공 기준이 문서화됨
- 구현 착수 전 Open Questions가 `open-questions.md`로 분리됨

## 11. 다음 단계

- 구현 대상 리포지토리 구조 분석
- 기술 스택 선택 문서 작성
- 개발 일정 수립
