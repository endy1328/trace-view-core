# Trace View Core PRD

## 1. 문서 목적

이 문서는 Trace View Core의 제품 요구사항을 정의한다. 대상은 기획, 설계, PoC, 우선순위 조정에 참여하는 개발 리드와 실무 담당자다.

## 2. 제품 개요

Trace View Core는 신규 인력이 특정 화면과 관련된 엔드포인트, 그리고 해당 엔드포인트 뒤에 있는 비즈니스 로직까지 빠르게 추적할 수 있도록 돕는 내부 분석 도구다.

핵심은 아래 흐름을 한 번에 보여주는 것이다.

- 화면
- 호출 엔드포인트
- 서버 진입점
- 핵심 서비스 또는 유스케이스
- 저장소 및 외부 연동

## 3. 문제 정의

현재 프로젝트 온보딩 과정에서 아래 문제가 발생한다.

- 화면에서 시작해 관련 API를 찾는 데 시간이 오래 걸린다.
- API에서 실제 비즈니스 로직 위치를 찾는 데 개인 역량 차이가 크다.
- 문서와 실제 코드가 자주 어긋난다.
- 변경 영향 범위를 빠르게 파악하기 어렵다.
- 신규 인력의 질문이 반복되고, 기존 인력의 설명 비용이 높다.

## 4. 목표

### 4.1 비즈니스 목표

- 신규 인력의 초기 적응 시간을 단축한다.
- 코드 이해를 특정 개인 의존에서 구조적 탐색 방식으로 전환한다.
- 변경 영향 분석 시간을 줄인다.

### 4.2 제품 목표

- 화면 기준으로 관련 엔드포인트를 1분 이내에 찾게 한다.
- 엔드포인트 기준으로 핵심 서버 로직 위치를 빠르게 찾게 한다.
- 화면에서 서버 로직까지 연결 관계를 시각적으로 제공한다.
- 설명과 메타데이터를 코드 탐색 결과에 덧붙일 수 있게 한다.

## 5. 비목표

초기 범위에서 아래는 목표로 잡지 않는다.

- 전사 시스템 전체를 한 번에 통합하는 것
- 100퍼센트 자동 정확도를 보장하는 것
- 배치, 메시지 큐, ETL, 레거시 시스템까지 한 번에 완전 분석하는 것
- 운영 모니터링이나 장애 대응 툴을 대체하는 것

## 6. 주요 사용자

### 6.1 신규 인력

주요 니즈:

- 화면 하나를 받아도 관련 API와 로직을 빠르게 찾고 싶다.
- 질문하기 전에 스스로 맥락을 파악하고 싶다.

### 6.2 기능 담당 개발자

주요 니즈:

- 특정 화면이나 API의 연결 구조를 빠르게 점검하고 싶다.
- 변경 영향이 어디까지 갈지 가늠하고 싶다.

### 6.3 리뷰어 및 리드

주요 니즈:

- 설계 의도와 실제 구현 위치를 빠르게 연결하고 싶다.
- 신규 인력에게 공통 탐색 기준을 제공하고 싶다.

### 6.4 운영 관리자

주요 니즈:

- 자동 분석 결과와 확정 메타데이터를 구분 관리하고 싶다.
- 검증 실패 문서와 미승인 요약을 통제하고 싶다.

## 7. 용어 정의

### 화면

MVP에서 화면은 사용자가 직접 진입 가능한 라우트 단위의 페이지를 기준으로 정의한다.
모달, 탭, 내부 섹션은 기본적으로 독립 화면으로 보지 않으며, 필요 시 하위 뷰로 메타데이터를 부여한다.

### 비즈니스 로직

MVP에서 비즈니스 로직은 API 진입점 이후 호출되는 서비스 또는 유스케이스 계층의 핵심 검증, 상태 변경, 외부 연동 처리 흐름을 의미한다.
순수 컨트롤러 바인딩 코드나 단순 DTO 변환 코드는 핵심 비즈니스 로직에서 제외한다.

## 8. 핵심 사용자 시나리오

### 시나리오 1. 화면에서 API 찾기

신규 인력이 `주문 상세 화면`을 검색한다.  
도구는 해당 화면의 라우트, 컴포넌트 위치, 호출 엔드포인트, 호출 순서, 주요 서비스 로직을 보여준다.

### 시나리오 2. API에서 핵심 로직 찾기

개발자가 `GET /orders/{id}`를 검색한다.  
도구는 엔드포인트를 처리하는 컨트롤러, 서비스 메서드, 저장소 접근, 외부 연동, 운영 메모를 보여준다.

### 시나리오 3. 변경 영향도 확인

리뷰어가 특정 서비스 메서드 변경 전, 연결된 화면과 상위 API를 확인한다.  
도구는 관련 화면, 엔드포인트, 하위 저장소 및 외부 호출을 연결 그래프로 보여준다.

## 9. 핵심 기능 요구사항

### 9.1 화면 중심 검색

사용자는 화면명, 라우트, 메뉴명으로 검색할 수 있어야 한다.

출력 정보:

- 화면 이름
- 라우트 경로
- 프론트 파일 위치
- 호출 엔드포인트 목록
- 호출 순서
- 대표 비즈니스 설명

### 9.2 엔드포인트 중심 검색

사용자는 HTTP method, path, 태그, 도메인 키워드로 검색할 수 있어야 한다.

출력 정보:

- 엔드포인트 정의
- 호출 화면 목록
- 서버 진입점 위치
- 서비스 호출 체인
- 저장소 접근
- 외부 시스템 연동

### 9.3 그래프 탐색

사용자는 아래 관계를 그래프로 볼 수 있어야 한다.

- 화면 -> API
- API -> 컨트롤러 또는 핸들러
- 핸들러 -> 서비스
- 서비스 -> 저장소
- 서비스 -> 외부 시스템

### 9.4 설명 보강

사용자는 화면, API, 서비스 노드에 메모 또는 설명을 추가할 수 있어야 한다.

예시:

- 핵심 검증 로직
- 주의할 예외 조건
- 배치 연관성
- 권한 또는 정책 의존성

### 9.5 영향도 탐색

사용자는 특정 노드를 기준으로 상하위 연결 노드를 탐색할 수 있어야 한다.

최소 제공 범위:

- 상위 호출 화면
- 중간 엔드포인트
- 하위 서비스, 저장소, 외부 호출

MVP에서는 연결 노드 탐색 중심으로 제공하고, 위험도 점수 계산은 후속 단계로 둔다.

### 9.6 자동 요약 승인 상태

비즈니스 로직 요약은 아래 상태를 가져야 한다.

- Draft: 자동 생성되었으나 사람 검토 전 상태
- Approved: 담당자 검토 후 확정된 상태
- Rejected: 부정확하거나 부적절하여 폐기된 상태

사용자는 자동 생성 요약과 확정 요약을 구분해서 볼 수 있어야 한다.

## 10. 비기능 요구사항

### 10.1 최신성

- 코드 분석 결과는 재실행 가능해야 한다.
- 문서와 결과는 수작업 문서만큼 쉽게 오래된 상태가 되면 안 된다.

### 10.2 탐색 속도

- 검색 응답은 실무 사용에 무리가 없을 정도로 빨라야 한다.
- 그래프 진입 시 핵심 노드부터 우선 노출해야 한다.

### 10.3 확장성

- 초기에는 한 프로젝트 기준으로 시작하되 멀티 서비스 확장 가능 구조여야 한다.
- 언어와 프레임워크 추가 지원이 가능해야 한다.

### 10.4 신뢰도

- 자동 추론 결과와 확정 메타데이터를 구분해서 보여줘야 한다.
- 사용자는 자동 분석 결과를 맹신하지 않도록 근거 위치를 확인할 수 있어야 한다.

### 10.5 보안 및 민감정보 처리

- 런타임 추적 데이터에서 민감한 헤더, 토큰, 개인정보, 세션 식별자는 저장 전에 마스킹 또는 제외해야 한다.
- 운영 메모와 주석은 권한 기반으로 접근 제어되어야 한다.
- 환경별 수집 정책을 분리해 운영 환경 데이터의 과수집을 막아야 한다.

## 11. MVP 범위

### 포함

- 대표 화면 20~30개 분석
- 프론트 1개 애플리케이션 연동
- 백엔드 1개 서비스 또는 BFF 연동
- 화면-API 매핑
- API-핸들러-서비스 체인 노출
- 기본 검색
- 기본 그래프
- 수동 주석

### 제외

- 멀티 리포지토리 통합
- 실시간 운영 모니터링
- 배치와 이벤트 흐름의 완전 추적
- 권한 정책 전체 시뮬레이션

## 12. 성공 지표

### 정량 지표

- 특정 화면의 관련 API 탐색 시간
- 특정 API의 핵심 서비스 로직 탐색 시간
- 온보딩 초기 질문 수
- 영향도 분석 소요 시간

측정 기준:

- 기준선 측정 대상: 신규 인력 3명 이상 또는 유사 역할 수행자
- 기준선 시점: 도구 도입 전 동일 과제 수행 시간
- 비교 시점: MVP 사용 후 동일 과제 재수행 시간
- 과제 예시: 특정 화면의 관련 API 찾기, 특정 API의 핵심 서비스 메서드 찾기
- 기록 방식: 시나리오별 시작 시각과 종료 시각 기록, 실패 여부 기록

### 정성 지표

- 신규 인력이 스스로 맥락을 파악할 수 있다는 체감
- 리뷰어가 연결 구조를 빠르게 확인할 수 있다는 체감
- 문서가 코드와 더 가깝다는 신뢰도

## 13. 운영 책임

- 문서 생성 담당: 초안 생성 및 요구사항 구조화
- 문서 검증 에이전트: 모순, 누락, 과도한 가정, 개선 대안 검토
- 메타데이터 승인자: 수동 주석과 승인된 비즈니스 요약 확정
- 운영 관리자: 검증 실패 문서와 접근 권한 관리

## 14. 우선순위

### P0

- 화면 검색
- API 검색
- 화면-API 연결
- API-백엔드 진입점 연결
- 서비스 체인 탐색

### P1

- 그래프 시각화
- 수동 주석
- 영향도 탐색
- 런타임 추적 연동

### P2

- 자동 요약 품질 고도화
- 멀티 프로젝트 확장
- 운영 메타데이터 연동

## 15. 리스크

- 정적 분석만으로는 동적 호출을 완전히 잡지 못할 수 있다.
- 공통 API 래퍼 구조가 복잡하면 화면-API 매핑 정확도가 떨어질 수 있다.
- 프레임워크별 차이로 분석기 구현 비용이 커질 수 있다.
- 비즈니스 설명 품질은 자동 생성만으로 한계가 있다.
- 운영 환경 데이터 수집 시 민감정보가 섞일 위험이 있다.

## 16. 대응 전략

- 정적 분석과 런타임 추적을 병행한다.
- 자동 결과와 확정 결과를 분리해 표시한다.
- 대표 화면군으로 PoC를 먼저 돌려 정확도와 운영 비용을 검증한다.
- 설명 계층은 수동 보강을 전제로 설계한다.
- 런타임 수집 데이터는 마스킹 정책과 환경별 수집 정책을 적용한다.

## 18. 문서 생성 및 검증 프로세스

Trace View Core 관련 기획 문서는 생성 단계와 검증 단계를 분리한다.

### 16.1 생성 단계

- 초안 작성 에이전트 또는 작성자가 PRD, 설계, 화면 문서를 생성한다.
- 생성 단계에서는 빠른 구조화와 내용 수집을 우선한다.

### 16.2 검증 단계

- 별도의 문서 검증 에이전트가 초안을 리뷰한다.
- 검증 에이전트는 아래를 점검한다.
  - 문서 간 논리적 모순
  - 누락된 요구사항 또는 운영 조건
  - 애매하거나 해석이 갈릴 표현
  - 구현 리스크가 큰 가정
  - 더 단순하거나 더 현실적인 대안

### 16.3 승인 조건

- 검증 에이전트가 치명적 모순이나 큰 누락이 없다고 판단해야 한다.
- 권장 개선사항은 별도 섹션으로 분리해 추후 반영 여부를 결정한다.
- 검증 결과와 반영 내역이 함께 남아야 한다.
- 아래 상태는 `문서 승인 상태`를 의미하며 트랙/작업 상태와는 구분한다.
- 문서 상태는 `Draft`, `In Progress`, `Review Pending`, `Validated`, `Approved`, `On Hold`, `Archived` 중 하나로 관리한다.

## 19. 출시 판단 기준

아래가 충족되면 MVP를 실사용에 올릴 수 있다.

- 대표 화면군에 대해 화면-API 매핑이 실무에서 쓸 만한 정확도를 보인다.
- 주요 API에 대해 진입점과 서비스 체인이 누락 없이 확인된다.
- 신규 인력 테스트에서 탐색 시간이 기존 방식보다 유의미하게 줄어든다.
