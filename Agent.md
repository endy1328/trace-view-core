# Trace View Core Agent Guide

## 목적

이 문서는 Trace View Core를 실제 개발 단계로 진행할 때, 멀티에이전트를 가장 빠르고 효율적으로 운영하기 위한 실행 규칙을 정의한다.

현재 제품 방향은 `Spring 기반 범용 분석 제품`이다.

이 문서는 아래 설계 문서를 기준으로 한다.

- `docs/master-plan.md`
- `docs/PRD.md`
- `docs/architecture.md`
- `docs/data-model.md`
- `docs/analysis-pipeline.md`
- `docs/operations.md`
- `docs/poc-plan.md`
- `docs/tech-stack-selection.md`

## 핵심 원칙

- 메인 에이전트는 항상 전체 우선순위와 통합 품질을 책임진다.
- 구현은 문서 순서가 아니라 `MVP 수직 슬라이스` 기준으로 진행한다.
- 각 서브 에이전트는 겹치지 않는 책임 범위를 가져야 한다.
- 문서와 코드 결과물은 항상 검증 에이전트가 별도로 리뷰한다.
- 지연이 발생하면 메인 에이전트는 추가 에이전트를 즉시 투입한다.
- 자동 생성 결과는 근거와 검증 상태 없이 확정하지 않는다.

## Spring 범용 제품 기준 개발 순서

Trace View Core는 아래 순서로 개발하는 것이 가장 효율적이다.

### 1. 구현 전 고정 결정

먼저 아래 3가지를 확정한다.

- Spring 샘플 또는 실제 대상 백엔드 코드베이스
- MVP에서 우선 지원할 Spring 표준 패턴 범위
- 기술 스택 선택 기준에 따른 구현 스택

이 단계의 목적은 범위를 고정해서 분석기와 UI를 과도하게 일반화하지 않게 하는 것이다.

### 2. Spring 코드베이스 구조 분석

Spring 코드 구조를 먼저 분석한다.

- 엔드포인트 선언 위치
- 서비스 계층 구조
- 저장소 및 외부 호출 패턴
- Bean 주입 구조

이 단계에서 실제 구현 난이도를 결정한다.

### 3. 공통 모델과 저장 구조 구현

가장 먼저 만드는 코드는 UI가 아니라 공통 모델이다.

- Screen
- ApiEndpoint
- BackendEntryPoint
- ServiceMethod
- Annotation
- Evidence
- Snapshot

이 모델이 먼저 고정되어야 이후 정적 분석기, 그래프, 검색, 검증이 모두 맞물린다.

### 4. Spring Core Analyzer MVP 구현

초기 MVP는 Spring 정적 분석 우선이다.

먼저 아래 연결만 되면 된다.

- ApiEndpoint -> BackendEntryPoint
- BackendEntryPoint -> ServiceMethod
- ServiceMethod -> RepositoryCall
- ServiceMethod -> ExternalCall

이 단계에서는 정확도 100퍼센트보다 `Spring 표준 프로젝트에서 실무 사용 가능 수준`이 더 중요하다.

### 5. 조회 API와 검색 구현

정적 분석 결과가 저장되면, 곧바로 검색과 상세 조회 API를 만든다.

- 화면 검색
- 엔드포인트 검색
- 화면 상세
- 엔드포인트 상세
- 서비스 체인 조회

그래프 UI보다 먼저 조회 API를 완성하는 게 효율적이다.

### 6. 최소 UI 구현

MVP UI는 아래만 우선 구현한다.

- 통합 검색 화면
- 엔드포인트 상세 화면
- 서비스 체인 상세 화면
- 기본 그래프 화면

주석 관리, 검증 결과, 스냅샷 비교는 2차로 붙인다.

### 7. 검증 흐름 구현

개발 초기에 같이 만들어야 하는 기능이다.

- 검증 상태 저장
- Must Fix / Recommended 분리
- Draft / Approved / Rejected 상태 관리
- 리뷰 이력 저장

이걸 나중에 붙이면 운영 모델이 깨진다.

### 8. Adapter Framework 구현

프로젝트별 특수 구조를 지원할 확장 포인트를 만든다.

- custom annotation adapter
- custom http client adapter
- custom repository adapter

### 9. 런타임 추적은 후속 보강

MVP에서는 개발/스테이징 환경만 대상으로 한다.

- 브라우저 호출 수집
- 백엔드 trace 연결
- 정적 분석 결과 보강

런타임 추적은 1차 구현이 아니라 2차 보강 트랙으로 운영한다.

## 멀티에이전트 운영 구조

## 1. 메인 에이전트

책임:

- 전체 계획과 우선순위 관리
- 병렬 작업 분해
- 결과 통합
- 충돌 조정
- 최종 품질 판단

규칙:

- 막히는 핵심 경로는 직접 처리한다.
- 서브 에이전트에는 독립적인 작업 단위만 넘긴다.
- 같은 파일을 여러 에이전트에게 동시에 수정하게 하지 않는다.

## 2. 코드 구조 분석 에이전트

책임:

- 백엔드 구조 분석
- Spring 표준 구조 식별

적합 작업:

- 컨트롤러/서비스 패턴 찾기
- Repository 패턴 찾기
- 외부 호출 패턴 찾기

## 3. 모델/백엔드 구현 에이전트

책임:

- 공통 모델 구현
- 저장 계층 구현
- 분석 결과 저장 API 구현

## 4. 분석기 구현 에이전트

책임:

- Spring Endpoint Analyzer 구현
- Spring Service Analyzer 구현
- Spring Repository / ExternalCall Analyzer 구현
- Mapping Engine 구현

## 5. API 구현 에이전트

책임:

- 검색 API
- 상세 조회 API
- 그래프 조회 API

## 6. UI 구현 에이전트

책임:

- 검색 화면
- 상세 화면
- 그래프 화면
- 검증 상태 UI

## 7. 검증 에이전트

책임:

- 코드 리뷰
- 문서와 구현 정합성 확인
- 테스트 누락 확인
- Must Fix / Recommended 분리

규칙:

- 구현 에이전트와 분리한다.
- 코드 결과물과 문서 결과물 모두 리뷰한다.

## 8. 상태 보고 에이전트

책임:

- 현재 진행률 집계
- 완료 / 진행중 / 지연 항목 보고
- blocker 정리

## 9. 지연 대응 에이전트

책임:

- 특정 트랙이 지연되면 즉시 추가 투입
- 분석, 구현, 테스트를 병렬 보조

규칙:

- 메인 에이전트가 지연 구간을 판단하면 즉시 생성
- 기존 에이전트와 겹치지 않는 범위만 맡긴다.

## 가장 빠른 병렬화 패턴

아래 패턴이 가장 효율적이다.

### Phase A. Spring 구조 파악 병렬화

- 에이전트 1: Spring endpoint/annotation 구조 분석
- 에이전트 2: Service/Repository 구조 분석
- 에이전트 3: 외부 호출/HTTP client 구조 분석

메인 에이전트는 이 결과를 합쳐 공통 모델과 구현 순서를 확정한다.

### Phase B. 기반 구현 병렬화

- 에이전트 1: 공통 모델 / DB 스키마
- 에이전트 2: 정적 분석기
- 에이전트 3: 조회 API

단, 파일 책임을 분리해야 한다.

### Phase C. 제품화 병렬화

- 에이전트 1: 검색/엔드포인트 상세 UI
- 에이전트 2: 그래프 UI
- 에이전트 3: 검증/주석/승인 흐름
- 에이전트 4: 테스트와 회귀 검증

## 에이전트 작업 분해 규칙

- 하나의 에이전트는 하나의 명확한 산출물만 책임진다.
- 하나의 작업은 1일 이하 단위로 쪼개는 것이 좋다.
- 파일 수정 범위가 겹치면 병렬화하지 않는다.
- 공통 모델 변경은 항상 메인 에이전트 승인 후 반영한다.
- PR 성격의 결과는 검증 에이전트 리뷰 전 확정하지 않는다.

## 개발 단계 문서 운영 규칙

- 구현 시작 전 해당 작업의 입력 문서를 명시한다.
- 구현 완료 후 어떤 문서를 충족했는지 연결한다.
- 설계 변경이 생기면 `docs/decision-log.md`를 먼저 갱신한다.
- 큰 변경은 리뷰 문서를 남긴다.

## 필수 체크리스트

각 작업 시작 전 아래를 확인한다.

- 이 작업은 어떤 설계 문서를 구현하는가
- 수정 파일 범위가 다른 에이전트와 겹치는가
- 출력 결과를 어떻게 검증할 것인가
- 상태 문서를 갱신해야 하는가

각 작업 완료 후 아래를 확인한다.

- 테스트가 있는가
- 근거 데이터가 저장되는가
- Draft / Approved 상태 모델을 위반하지 않는가
- 민감정보 정책을 어기지 않는가

## 검증 우선순위

검증 에이전트는 아래 순서로 본다.

1. 문서와 구현의 불일치
2. 공통 모델 위반
3. 근거 저장 누락
4. 상태 모델 위반
5. 보안 정책 위반
6. 테스트 누락
7. 성능 문제

## 상태 보고 규칙

상황파악 에이전트는 아래 형식으로만 보고한다.

- 현재 단계
- 완료 작업
- 진행 중 작업
- 지연 작업
- 추가 에이전트 투입 여부
- 다음 액션

## 개발 착수 추천 순서

실제 시작은 아래 순서가 가장 효율적이다.

1. Spring 샘플 또는 실제 대상 코드베이스 구조 분석
2. 기술 스택 선택 확정
3. 공통 데이터 모델 구현
4. Spring Core Analyzer MVP 구현
5. 검색/상세 API 구현
6. 최소 UI 구현
7. 검증/승인 흐름 구현
8. Adapter Framework 구현
9. 런타임 추적 보강

## 금지 사항

- 처음부터 멀티 리포지토리 일반화를 시도하지 않는다.
- 런타임 추적을 MVP 1순위로 올리지 않는다.
- 자동 요약을 승인 없이 기본 설명으로 노출하지 않는다.
- 검증 에이전트 없이 결과를 확정하지 않는다.
- 지연이 생겼는데 단일 에이전트로 오래 끌지 않는다.

## 결론

Trace View Core 개발은 `Spring Core Analyzer를 빠르게 세우고, 검색/상세 흐름을 먼저 완성한 뒤, 검증과 운영 모델을 함께 붙이고, 프로젝트별 차이는 Adapter로 분리하고, 런타임 추적은 후속 보강`으로 가는 것이 가장 효율적이다.

멀티에이전트 운영은 `구조 분석`, `기반 구현`, `제품 UI`, `검증`, `상태 보고`를 분리하고, 병목 구간마다 추가 에이전트를 투입하는 방식으로 운영한다.
